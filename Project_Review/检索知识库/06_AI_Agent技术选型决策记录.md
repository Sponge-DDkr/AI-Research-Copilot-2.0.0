# AI Agent 技术选型决策记录

> 决策日期：2026-06-01 | 决策者：架构组（孙架构、钱后端、赵前端）
> 决策类型：架构决策 | 状态：已执行

---

## 决策背景

AI Research Copilot 项目启动前，需要确定 Agent 编排框架。市场上有多个选项，各有利弊。

---

## 候选方案

### 选项 1：LangChain + LangGraph

**简介**：Python AI 框架的事实标准，LangGraph 提供了 StateGraph 循环能力。

**优势**：
- 生态最丰富（200+ integrations，100+ 内置工具）
- 社区活跃（GitHub 90k+ stars）
- 文档相对完善
- 面试认可度最高

**劣势**：
- 抽象层太厚：`RunnablePassthrough` / `RunnableLambda` / `RunnableSequence` 等概念学习成本高
- 版本迭代快但不稳定：0.1 → 0.2 → 0.3 每次升级有破坏性变更
- 隐藏了太多细节：Agent 循环的具体行为对开发者不透明
- 对于我们的简单场景（10 个工具、单 Agent Loop），LangGraph 是杀鸡用牛刀
- 序列化限制：所有 State 必须可序列化（JSON），限制了工具返回复杂对象

### 选项 2：自研 Unified Agent Loop ✅（最终选择）

**简介**：200 行 Python 实现 ReAct + Plan-and-Execute 混合循环。

**优势**：
- **完全透明**：每一行代码都是自己写的，出问题能定位到具体行
- **极致可控**：Stop Gate、复杂度判断、工具过滤、记忆注入等定制逻辑天然集成
- **没有黑盒**：不依赖第三方框架的版本变更
- **学习价值**：自己实现一遍，面试时能把原理讲透
- **轻量**：0 额外依赖（除了 httpx 调 LLM API）

**劣势**：
- 功能不全：没有 LangSmith 那样的调试追踪
- 需要自己维护和演进
- 没有社区支持
- 工具生态需要自己搭建

### 选项 3：CrewAI / AutoGen 多 Agent 框架

**简介**：定义多个 Agent 角色，让它们自动协作。

**劣势（一票否决）**：
- Agent 之间的通信靠 LLM 生成的自然语言，一个研究任务可能要 30-50 次 LLM 调用
- 成本不可控：一次简单搜索可能触发 Agent 之间的「无限对话」
- 不确定性强：同样的输入输出差异大
- 不适合生产环境：更适合实验和 Demo

## 最终决策：选项 2 —— 自研 Unified Agent Loop

### 核心理由

> "理解原理比使用框架更重要。自己实现一遍 Agent Loop，200 行代码换来的理解，比跟着 LangChain 教程跑 10 遍更深。面试时，能讲清楚 'Agent 循环的每一步发生了什么' 比 '我用过 LangGraph' 更能证明技术深度。"
>
> — 孙架构，决策会议上的总结

### 决策后的补充说明

这个决策**不等于否定 LangChain**。在以下场景中，我们仍然会使用 LangChain：
- 需要快速接入某小众 LLM 提供商时（LangChain 的 `ChatModel` 抽象屏蔽差异）
- 文档加载器（`PyPDFLoader` / `UnstructuredMarkdownLoader`）——没必要重复造轮子
- 评估框架（LangSmith 的 Tracing 能力在调试时很有用）

**核心原则**：核心逻辑（Agent Loop、工具调度、记忆管理）自己掌控。周边设施（文档加载、评估）可以用现成的。

## 后续演进计划

如果项目发展顺利（超过 20 个工具 / 需要多人协作），会考虑：
1. 将自研 Loop 抽象为内部框架，定义 Standard Tool Interface
2. 引入 LangGraph 但只用于复杂分支流程（如多 Agent 辩论模式），简单路径仍是自研
3. 不自研的工具部分（文档加载）继续用 LangChain 组件

### 4. 为什么不选 LangChain 的 7 条详细理由

**理由 1：版本破坏性变更频繁且缺乏平滑迁移路径**
LangChain v0.1→v0.2→v0.3 每次大版本升级都引入了破坏性 API 变更。我们团队内部有 3 个项目因 LangChain 0.2→0.3 的 StateGraph API 签名变化导致延期 2 周。`add_condition_edges` 的参数从 `(edges: Dict[str, str])` 改为 `(paths: List[Edge])`，看似简单的改动需要重写所有条件分支逻辑。

**理由 2：抽象层过厚导致调试困难**
LangChain 的 `RunnablePassthrough` / `RunnableLambda` / `RunnableSequence` / `RunnableBranch` 构成的链式抽象链使得错误栈难以追踪。一次工具调用失败的错误栈平均包含 12 层框架内部调用，定位根本原因需要理解框架的内部实现细节——这与"你应该关注业务逻辑，不用关心框架内部"的初衷相悖。

**理由 3：框架序列化约束限制了工具能力**
LangChain 要求所有 State 必须是可序列化的（JSON-compatible），这意味着工具不能返回 Pydantic 对象、numpy 数组、PIL Image 等复杂类型。我们有一个需要返回 `matplotlib.Figure` 的分析工具，在 LangChain 中需要额外实现自定义序列化器，而在自研方案中只需要直接返回对象引用。

**理由 4：框架 Prompt 注入导致 Token 效率更低**
LangChain 在每个 Agent 循环中会注入多条模板 Prompt（系统 Prompt、格式说明 Prompt、记忆 Prompt、工具描述 Prompt）。实测发现，同样的任务 LangChain 比自研方案多消耗约 18-22% 的 Token。对于每月数百万 tokens 的调用量，这部分额外成本不可忽视。

**理由 5：社区驱动导致的碎片化问题**
LangChain 拥有 200+ 集成组件，但质量参差不齐。部分社区维护的集成（如 `WebResearchRetriever`、`WikipediaQueryRun`）文档不完善且测试覆盖率低。我们在使用 `WikipediaQueryRun` 时遇到了 Unicode 编码 bug，修复 PR 在社区等待 3 周才被合并——这对于需要快速迭代的 0-1 阶段项目来说不可接受。

**理由 6：对工具调用流程的不透明控制**
Agent 的 Stop Gate、工具过滤、参数校验、重试策略在 LangChain 中需要继承多个抽象类（`BaseTool`、`BaseCallbackHandler`、`BaseChatMessageHistory`）并重写回调方法。自研方案中，一个不超过 50 行的 `_execute_tool()` 函数就能承载全部定制逻辑，且不需要理解框架的回调机制。

**理由 7：Lock-in 风险不符合长期架构策略**
将 Agent 核心逻辑绑定在 LangChain 上意味着未来迁移到其他框架（或自研）时需要重写全部 Agent 逻辑。而选择将核心 Loop 自研 + 仅在周边组件（文档加载、评估追踪）复用 LangChain，确保了未来可以在不影响核心逻辑的前提下更换周边设施。这是架构层面的"依赖倒置"——不应该让框架定义你的核心逻辑架构。

### 5. 2026 下半年 Agent 框架趋势判断

**MCP 协议标准化对 Agent 框架的影响**

模型上下文协议（Model Context Protocol，MCP）正在成为 Agent 工具调用的标准化协议。截至 2026 年 7 月，Anthropic 已发布 MCP 规范 v1.0，OpenAI 已在其 Agents SDK 中支持 MCP 兼容模式。对于我们项目的影响：
- 短期（2026 Q3）：监控 MCP 生态成熟度，在工具接口层预留 MCP 适配器接口
- 中期（2026 Q4）：如果 MCP 成为事实标准，将工具注册机制改为 MCP 协议兼容
- 长期（2027+）：MCP 可能取代当前各种框架自有的工具注册机制，成为 Agent 工具的"HTTP 协议"

当前策略：不允许将工具注册直接绑定在框架 API 上，必须通过内部抽象层隔离。即使未来切换到 MCP 协议，也只需要替换适配层。

**OpenAI Agents SDK 的竞争威胁**

OpenAI 于 2026 年初发布的 Agents SDK 提供了开箱即用的 Agent 编排能力，深度集成 GPT-4o/5 系列模型。主要优势：
- 零配置 Agent 编排
- 内置 Guardrails（内容安全、格式验证）
- OpenAI 模型的原生优化（调用延迟降低 30%）
- 流式工具调用支持

**威胁评估**：如果 SDK 在 2026 Q4 前支持非 OpenAI 模型，将对 LangGraph 和自研方案的生态位形成直接冲击。我们的应对策略是确保核心 Loop 设计不与特定 LLM 提供商绑定——目前已经使用了 `BaseLLMClient` 抽象，允许在 2 天内切换提供商。

**Google ADK 的企业市场优势**

Google 的 Agent Development Kit（ADK）专注于企业级需求：
- GCP 原生集成（Vertex AI、BigQuery、Cloud Storage）
- 企业 IAM 权限模型深度集成
- 合规证书齐全（SOC2、ISO 27001、FedRAMP）
- 支持多模态 Agent（文本+图像+代码）

**适用场景**：如果项目未来需要进入金融、医疗等受监管行业，Google ADK 的企业合规能力会成为关键优势。但 ADK 目前锁死在 GCP 生态内，对于多云策略的发展方向不友好。

**本项目应对策略**：

| 趋势 | 应对策略 | 时间节点 |
|------|---------|---------|
| MCP 标准化 | 在工具注册层增加 MCP 适配器接口 | 2026 Q3 完成设计，Q4 完成实现 |
| OpenAI Agents SDK | 保持 LLM 无关的核心设计，不依赖特定 SDK | 已实现 |
| Google ADK 企业需求 | 仅在进入受监管行业时评估 ADK 切换成本 | 2027 Q1 评估 |
| 自研 Loop vs 框架 | 维持"核心自研 + 周边复用"策略，每季度评估一次是否改选框架 | 每季度初评估 |
| 多 Agent 模式 | 如果工具数超过 50 个，评估是否引入 LangGraph 的子图编排 | 触发条件时评估 |

**核心原则不变**：不允许将 Agent 核心逻辑绑定在任何特定框架上——这是 2026 年 6 月决策会议上的架构决定，本补充数据不改变该决定。
