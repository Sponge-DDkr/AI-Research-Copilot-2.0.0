## 一、总则

### 1.1 目的

本规范定义公司 AI 应用系统的统一架构标准，确保各业务线 AI 项目在**安全性、可扩展性、可观测性**三个维度达到基线要求。

### 1.2 适用范围

- 所有调用 LLM API 的后端服务
- 所有使用向量数据库的应用
- 所有面向用户的 AI Chatbot / Agent 产品
- 内部 AI 工具链（Copilot、代码审查、测试生成）

## 二、核心技术栈标准

### 2.1 后端框架

强制使用 **Python 3.12+** + **FastAPI 0.115+**。新项目禁止使用 Flask 或 Django。

理由：FastAPI 原生 async、自动 OpenAPI 文档、Pydantic 深度集成、SSE 支持成熟。

### 2.2 LLM 调用规范

```
模型优先级：
  Tier 1: DeepSeek V3（主力，中文 + 成本）
  Tier 2: Claude Sonnet 4（高难度推理、质量把关）
  Tier 3: GPT-4o（备选，多模态需求）

调用约束：
  - 所有 LLM 调用必须带 timeout（默认 30s，最长 120s）
  - 必须实现 retry（exponential backoff，最多 3 次）
  - 请求/响应必须记录到日志系统（脱敏后）
```

### 2.3 向量数据库选型

| 场景 | 推荐方案 | 备注 |
|------|---------|------|
| 本地开发 / Demo | ChromaDB 0.6+ | 零配置，Python 原生 |
| 生产环境 < 100 万向量 | Qdrant | 性能优秀，过滤强大 |
| 生产环境 > 100 万向量 | Milvus 分布式 | 水平扩展 |
| 已有 ES 集群 | Elasticsearch 8.x | 复用基础设施 |

**向量维度标准**：1024（BGE-M3 或 text-embedding-3-large）

### 2.4 嵌入模型规范

**当前标准模型**：`BAAI/bge-m3`（1024 维，支持 dense + sparse 双输出）

切换模型时必须：
1. 创建新 Collection（命名 `{name}_v{model_version}`）
2. 后台异步重建索引
3. 验证 MRR ≥ 旧索引的 95%
4. 热切换 alias
5. 保留旧索引 7 天用于回滚

## 三、安全规范

### 3.1 API 密钥管理

**绝对禁止**：
- API Key 硬编码在源代码中
- API Key 提交到 Git 仓库
- API Key 打印到日志

**强制要求**：
- 使用环境变量（`.env`）+ 密钥管理服务（Vault/Berglas）
- `.env` 文件必须在 `.gitignore` 中
- 生产环境使用 K8s Secret 或云厂商 Secret Manager

### 3.2 用户数据保护

- 用户输入默认保留 30 天，超期自动清除
- PII（手机号、身份证、邮箱）自动脱敏：`138****1234`
- 用户可随时请求删除所有个人数据（48 小时内执行）

### 3.3 Agent 安全约束

```
Agent 工具调用黑名单（绝对不可执行）：
  - 删除文件/数据库记录
  - 修改系统配置
  - 发送外部 HTTP 请求到非白名单域名
  - 执行任意 Shell 命令
  - 访问 /etc、/proc、环境变量

Agent 工具调用白名单（限定范围内可用）：
  - 搜索（仅限 Tavily API / 内网 Elasticsearch）
  - 读写知识库（仅限指定 Collection）
  - 生成报告（仅限指定模板）
```

## 四、性能基线

### 4.1 响应时间要求

| 端点类型 | P50 | P95 | P99 |
|---------|-----|-----|-----|
| 简单问答（无工具） | < 2s | < 5s | < 8s |
| Agent 研究（浅度） | < 15s | < 30s | < 45s |
| Agent 研究（深度） | < 60s | < 120s | < 180s |
| 知识库上传 | < 5s/MB | < 10s/MB | < 20s/MB |

### 4.2 并发要求

- 单实例支撑 50 并发 SSE 连接
- 向量检索 P95 < 100ms
- LLM API 调用超时 30s（可配置）

## 五、RAG 质量评估基线

### 5.1 评估指标

| 指标 | 基线 | 目标 |
|------|------|------|
| Recall@5 | ≥ 0.75 | ≥ 0.85 |
| MRR | ≥ 0.60 | ≥ 0.75 |
| Faithfulness | ≥ 0.80 | ≥ 0.90 |
| Answer Relevance | ≥ 0.70 | ≥ 0.85 |

### 5.2 评估频率

- 每次模型/参数变更后必须跑评估
- 每周自动跑全量 Golden Dataset（当前 200 条）
- 低于基线 → 告警通知架构组 → 24 小时内修复或回滚

### 5.3 Golden Dataset 管理

Golden Dataset 存储在 `internal/eval/golden_dataset.jsonl`，每季度由架构组扩充 50 条。每条包含：query、expected_chunks、min_score。

## 六、模型灰度发布策略（v3.2 新增）

### 6.1 灰度阶段

```
阶段 0: 内部测试（架构组 + QA，10 人，1 天）
阶段 1: 1% 流量（真实用户，观察 4 小时）
阶段 2: 10% 流量（观察 24 小时）
阶段 3: 50% 流量（观察 48 小时）
阶段 4: 100% 全量

每个阶段的回滚条件：
  - 错误率 > 基线的 2 倍
  - P95 延迟 > 基线的 1.5 倍
  - Faithfulness < 0.75
  - 用户投诉 > 5 条/小时
```

### 6.2 模型版本记录

当前运行模型：
- 主力：`deepseek-chat`（2026-06-15 灰度完成）
- 审核：`claude-sonnet-4-20250514`（2026-06-10 灰度完成）

### 附录 E：API Key 管理细则补充

**Key 轮换策略**
- 轮换周期：**90 天**（强制更换，到期前 7 天钉钉/邮件通知负责人）
- 轮换窗口：每周六凌晨 02:00-04:00（低峰期）
- 新旧 Key 并行期：最多 24 小时（用于平滑迁移）

**泄露应急流程（3 步）**
1. **吊销** — 发现泄露后 5 分钟内从 Secret Manager 吊销 Key，阻断继续使用
2. **审计** — 拉取泄露前 24 小时的调用日志，审查异常访问模式（异常 IP、异常用量、非常规时段）
3. **通知** — 通知安全组 + 产品负责人，填写安全事件报告（模板见 `internal/security/incident_report.md`）

**不允许的操作清单（严禁违反）**
- 禁止将 API Key **硬编码**在源代码中（包括常量文件、配置文件、测试文件）
- 禁止将 API Key 提交到 **Git 仓库**（包括 .env.example 中的真实值、历史 commit 中的残留）
- 禁止 **跨环境复用** Key（开发环境与生产环境必须使用不同 Key）
- 禁止在任何日志中 **打印** API Key（包括调试日志、错误堆栈、审计日志）
- 禁止将 Key 写入临时文件、缓存目录或 HTML 源代码
- 禁止通过未加密通道传输 Key（如明文 HTTP、IM、邮件正文）

### 附录 F：灰度发布各阶段时长与回滚条件表

| 阶段 | 流量比例 | 持续时间 | 回滚判断标准 |
|------|---------|---------|------------|
| 阶段 0：内部测试 | 仅架构组 + QA（10人） | **2 小时** | 任何功能阻塞 Bug；Faithfulness < 0.75 |
| 阶段 1：小流量 | 1% 真实用户 | **4 小时** | 错误率 > 基线 2 倍；P95 延迟 > 基线 1.5 倍 |
| 阶段 2：中流量 | 10% 真实用户 | **24 小时** | 错误率 > 基线 1.5 倍；用户投诉 > 5 条/小时 |
| 阶段 3：大流量 | 50% 真实用户 | **48 小时** | 错误率 > 基线 1.2 倍；NPS 下降 > 10 点 |

**回滚操作规范**
- 回滚决策由架构组值班人判定，**不允许**开发人员自行回滚
- 回滚时间窗口：发现触发条件后 **15 分钟内**必须执行回滚操作
- 回滚后自动触发告警通知：「架构组 + QA + 产品负责人」
- 除非经过架构评审，否则同一次灰度发布**不允许**连续回滚超过 2 次

### 附录 G：Agent 安全约束黑名单补充

以下 **8 类系统命令** Agent 在任何场景下禁止执行：

| 编号 | 命令类别 | 示例 | 风险等级 |
|------|---------|------|---------|
| 1 | 危险删除 | `rm -rf /`、`rm -rf /*`、`shred` | 🔴 致命 |
| 2 | 权限绕过 | `chmod 777 /`、`chown` 提权 | 🔴 致命 |
| 3 | 远程执行 | `curl <url> \| bash`、`wget -O- \| sh` | 🔴 致命 |
| 4 | 提权操作 | `sudo`、`su`、`pkexec` | 🟠 高危 |
| 5 | 强制终止 | `kill -9 <pid>`（不允许，必须用 `kill -15`） | 🟠 高危 |
| 6 | 挂载操作 | `mount`、`umount`、`swapon` | 🟡 中危 |
| 7 | 网络配置 | `iptables`、`ip link`、`ifconfig down` | 🟡 中危 |
| 8 | 容器管理 | `docker rm -f`、`docker rmi`、`docker system prune -a` | 🟠 高危 |

> **例外机制**：如需临时使用上述命令（如应急恢复场景），必须通过 MCP 工具的 `admin_override` 接口申请，经架构组值班人审批后获得一次性 Token 授权（有效期 30 分钟，超期自动失效）。
