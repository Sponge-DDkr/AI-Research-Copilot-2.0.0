## 二、Q3 完成的重点项目

### 项目 A：智能报告生成引擎升级（代号「笔杆子」）

- 上线时间：2026-08-15
- 负责人：王前端 + 钱后端
- 核心改动：从固定模板升级为 LLM 动态生成，支持 12 种报告类型
- 用户反馈评分：4.7/5.0
- **关键决策**：放弃了自研模板引擎，改用 LLM + JSON Schema 约束输出格式。理由是自研模板维护成本太高（历史上 12 种报告类型 × 3 个版本的排列组合导致 36 套模板的维护噩梦）。
- 技术细节：使用了 DeepSeek V3 的 JSON Mode，配合 Pydantic 做输出校验，不合规的输出自动重试（最多 2 次）。

### 项目 B：Agent Timeline 可视化重构

- 上线时间：2026-09-05
- 负责人：赵前端
- 核心改动：用 SSE 事件驱动的时间线组件替代旧的静态进度条
- 用户满意度从 3.2 提升到 4.6
- 技术栈：React 19 + TailwindCSS 4 + `react-markdown`
- **最大的坑**：SSE 断连重试。最初用 `EventSource` API，但它不支持 POST + 自定义 header。后来改为 `fetch` + `ReadableStream` + 手动行解析，断连后 3 秒自动重连。

### 项目 C：企业知识库权限系统

- 上线时间：2026-09-28
- 负责人：孙后端
- 支持按部门/角色/个人三级权限控制
- 技术方案：每个文档 chunk 的 metadata 里带 `access_level` 标签（`public` / `team` / `dept` / `private`），检索时按用户权限做 metadata filter
- **安全审计**：通过外部安全公司渗透测试，0 高危漏洞

## 三、未达标的项目

### 项目 D：PDF 复杂表格解析

- 原计划 Q3 完成，延迟到 Q4
- 根因：非结构化 PDF（扫描件、合并单元格）的表格解析准确率只有 72%，不满足 95% 的基线要求
- 尝试了 3 种方案：PyMuPDF → Camelot → Unstructured.io → 最终选定 Azure Document Intelligence（准确率 96%，但成本 ¥0.15/页）
- **教训**：不要高估开源工具对中文复杂排版的解析能力。预算允许的情况下，付费 API 省的时间远超成本。

### 十一、Q4 明确不做清单（放弃/禁止/不允许）

以下 4 项功能已由产品委员会讨论决定，**不允许**在 Q4 开发：

1. **长报告生成限制**：AI 助手在聊天模式中**不允许**生成超过 **2000 字**的完整报告。超过此限制的报告必须引导用户切换至「深度研究」模式。理由是聊天模式的 Context Window 不适合长文本生成，容易导致上下文漂移和质量下降。

2. **Markdown 实时协作编辑**：**不会支持**多人实时协作编辑 Markdown 文档（如 Google Docs 风格）。除非用户提出明确需求且经产品委员会投票通过（≥ 3/5 票）。理由是当前用户群体中单人编辑占比 96%，协作编辑的工程复杂度不匹配当前收益。

3. **日志脱敏规则**：**禁止**在未脱敏的日志中打印任何用户输入内容。所有日志输出必须经过 PII 脱敏管道处理（手机号、邮箱、身份证号、API Key 等）。包括调试日志、错误堆栈、审计追踪在内，**一律不允许**记录原始用户输入。

4. **自研 Embedding 模型**：**放弃**自研 Embedding 模型的计划。除非 HuggingFace **MTEB 中文榜单**中出现新模型在核心指标（Recall@5、MRR、STS）上超越 `BAAI/bge-m3` 或 `BGE-large-zh-v1.5` 超过 **10%** 以上，否则不考虑投入自研资源。理由是自研 Embedding 模型的投入产出比远低于优化现有 RAG Pipeline。
