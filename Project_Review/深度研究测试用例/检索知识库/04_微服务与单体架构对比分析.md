# 微服务 vs 单体架构：AI 应用的架构选型分析

> 作者：孙架构 | 日期：2026-06-15 | 状态：决策已定

---

## 背景

公司在 Q2 末启动了 AI Research Copilot 项目。技术选型阶段，架构组内对「微服务还是单体」产生了激烈争论。本文档记录完整的分析过程和最终决策。

---

## 一、两个方案的详细定义

### 方案 A：微服务架构

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ Agent    │  │ Search   │  │ Write    │  │ Review   │
│ Service  │  │ Service  │  │ Service  │  │ Service  │
│ (Port    │  │ (Port    │  │ (Port    │  │ (Port    │
│  8001)   │  │  8002)   │  │  8003)   │  │  8004)   │
└────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘
     │             │             │             │
     └─────────────┼─────────────┼─────────────┘
                   │             │
              ┌────┴────┐   ┌────┴────┐
              │ Redis   │   │ PostgreSQL│
              └─────────┘   └──────────┘
```

- 每个 Agent 角色（Search / Analyze / Write / Review）是独立的微服务
- 服务间通过 gRPC 通信
- 每个服务有独立的数据库
- K8s 部署，独立扩缩容

### 方案 B：单体架构（最终选择 ✅）

```
┌─────────────────────────────────────────┐
│              FastAPI 单体应用             │
│  ┌──────────────────────────────────┐   │
│  │        Unified Agent Loop        │   │
│  │  LLM → 选择工具 → 执行 → 观察    │   │
│  └──────────────────────────────────┘   │
│  ┌──────────┐ ┌──────────┐ ┌─────────┐ │
│  │ 搜索工具  │ │ 分析工具  │ │ 写作工具 │ │
│  └──────────┘ └──────────┘ └─────────┘ │
│  ┌──────────────────────────────────┐   │
│  │   SQLite + ChromaDB（同进程）     │   │
│  └──────────────────────────────────┘   │
└─────────────────────────────────────────┘
```

- 所有功能在一个 FastAPI 进程里
- 不同 Agent 角色以「工具」形式注册，不是独立服务
- SQLite + ChromaDB 本地运行

## 二、多维度对比

### 2.1 开发效率

| 维度 | 微服务 | 单体 |
|------|--------|------|
| 初始搭建时间 | 2-3 天（K8s + CI/CD + 服务脚手架） | 2 小时 |
| 本地调试 | 需要同时起 4+ 个服务 | `uvicorn main:app` 一行 |
| 新增 Agent 能力 | 新建服务 + 注册 gRPC + 配 K8s | `@registry.register()` 装饰器，10 行代码 |
| 代码理解成本 | 高（跨服务调试困难） | 低（状态在同一个 AgentState 里） |

### 2.2 性能

微服务在「独立扩缩容」方面有优势，但我们的场景里**瓶颈在 LLM API 调用（IO 等），不在计算**。

实测数据（同机，100 并发 SSE 连接）：
- 微服务架构（gRPC 通信）：P95 延迟 **18.7s**
- 单体架构（函数调用）：P95 延迟 **14.2s**

微服务反而慢 **24%**。原因是 gRPC 序列化/网络往返的额外开销。Agent 每次工具调用如果走 gRPC（而非函数调用），会增加 15-30ms 的额外延迟。一个深度研究任务平均 10 次工具调用，累积额外延迟 150-300ms。

### 2.3 团队规模适配

> 这个项目的开发团队是 **2-3 人**。微服务的运维成本（K8s、服务发现、链路追踪、日志聚合）需要至少 5 人的平台工程团队支撑。在 2-3 人团队里引入微服务，等于把有限的人力从「开发业务功能」转移到了「维护基础设施」——这是本末倒置。

### 2.4 未来迁移成本

当前选择单体，不代表永远单体。SQLAlchemy 2.0 ORM 支持一行配置从 SQLite 切换到 PostgreSQL。如果未来用户量增长需要拆分服务，也只需要拆成 3 个服务而非 10 个：

```
未来拆分方案（如果需要）：
  Service 1: Agent Engine（核心 Loops + 工具调度）
  Service 2: Knowledge Service（知识库管理 + 检索）
  Service 3: API Gateway（用户认证 + 限流 + 路由）
```

## 三、最终决策

### 决策：方案 B（单体架构）

**为什么选 A 不选 B？** 文档开头故意写反了。微服务才是方案 A。我们选了单体（方案 B）。

5 条理由，按重要性排序：

1. **团队规模不匹配**：2-3 人的团队维护微服务是资源浪费
2. **瓶颈在 LLM API 而非计算**：独立扩缩容的优势在 IO 密集型场景不存在
3. **Agent 的 Plan State 是共享状态**：微服务间拆分会破坏状态的连贯性——Agent 的 LLM 需要「看到所有历史工具调用结果」，拆成服务后要么传巨大的上下文（网络开销），要么各服务自己维护不完整的状态
4. **实测数据不支持微服务**：gRPC 通信的额外延迟让性能反而下降了 24%
5. **KISS 原则**：项目阶段是 0-1，不是 1-N。0-1 阶段需要快速迭代验证假设，微服务的复杂度会拖慢迭代速度

### 不选微服务的「反后悔药」

如果未来出现以下任一情况，立即启动微服务拆分：
- 单一团队超过 15 人
- 知识库检索成为独立产品（有独立的业务逻辑和团队）
- 日活超过 10 万，需要异地多活部署

---

## 四、经验总结

> **架构选型的第一原则：不是「什么架构最好」，是「什么架构最适合当前团队和当前阶段」。**
>
> 微服务不是银弹。Martin Fowler 在 2015 年提出的「Monolith First」原则到今天依然成立——先用单体验证产品，找到真正的业务边界后，再有针对性地拆分。不要为了「面试时可以讲微服务」而做微服务。
