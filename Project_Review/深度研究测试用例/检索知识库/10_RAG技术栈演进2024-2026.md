# RAG 技术栈演进 2024-2026：从 Naive RAG 到 Agentic RAG

> 作者：小MO | 日期：2026-07-18 | 状态：持续更新中

---

## 一、三阶段演进概述

RAG（Retrieval-Augmented Generation）技术在 2024-2026 年间经历了三次显著的技术跃迁。每一阶段都解决了前一阶段的核心瓶颈，同时也引入了新的挑战。

```
时间线：
  Naive RAG (2023初 - 2024初)  →  Advanced RAG (2024 - 2025)  →  Agentic RAG (2025 - 2026)
      原始向量检索                   混合检索 + 重排序                LLM 自主决策检索策略
      Top-K 硬匹配                  多路融合 + 查询改写               多步推理 + 自我验证
      高幻觉率 15-25%               幻觉率降至 5-10%                  幻觉率 < 3%
```

### 1.1 Naive RAG（2023 初 - 2024 初）

**定义**：最早期的 RAG 实现范式，将文档切块后 Embedding 入库，检索时用用户问题 Embedding 做向量相似度搜索，取 Top-K 结果直接拼入 Prompt 交给 LLM。

**代表技术**：LangChain RetrievalQA、LlamaIndex SimpleRetriever、OpenAI Embeddings + ChromaDB

**核心局限**：
- Chunk 切分粗糙（固定 512 token 切分，不考虑语义边界）
- 单路检索（仅 Dense Embedding，无 Sparse/BM25 互补）
- 无排序优化（直接返回向量距离排序，不做 Rerank）
- 查询无改写（用户口语化问题直接用于检索，与文档语义不匹配）
- 幻觉率高（15-25%，检索不相关时 LLM 强行生成）

### 1.2 Advanced RAG（2024 - 2025）

**定义**：在 Naive RAG 基础上引入检索前/检索后/检索中三大优化环节，系统性地提升检索质量和生成质量。

**代表技术**：混合检索（Dense + Sparse）、RRF 融合排序、Cross-Encoder Reranker、Query Rewriting、HyDE、句子窗口检索、元数据过滤

**核心突破**：幻觉率从 15-25% 降至 5-10%，Recall@5 从 0.45-0.58 提升至 0.70-0.85

### 1.3 Agentic RAG（2025 - 2026）

**定义**：LLM 不再是被动接收检索结果的生成器，而是主动决策检索策略的「检索 Agent」。模型自主判断：是否需要检索、检索什么、用什么工具检索、检索结果怎么用。

**代表技术**：Self-RAG、Corrective RAG (CRAG)、Adaptive RAG、FLARE、ReAct + RAG

**核心突破**：检索成为 Agent 的工具之一，与代码执行、API 调用、Web 搜索等工具协同，实现多步推理和自我验证。幻觉率进一步降至 < 3%。

## 二、Naive RAG 架构详解

### 2.1 标准流程

```
用户输入
    │
    ▼
Embedding 模型（text-embedding-ada-002 / bge-small-zh）
    │
    ▼
向量数据库（ChromaDB / FAISS）→ Top-K（通常 K=4 或 K=5）
    │
    ▼
原始 Chunk 直接拼入 Prompt
    │
    ▼
LLM 生成回答（GPT-4 / Claude / 文心一言）
```

### 2.2 主要局限与数据

| 问题 | 具体表现 | 量化数据 |
|------|---------|---------|
| Chunk 切分粗糙 | 固定 512 token 切分，打断段落/句子 | 信息丢失率约 18-22% |
| 单路检索 | 仅 Dense Embedding，对专有名词/缩写不敏感 | 专有名词检索 Recall 下降 35% |
| 无 Rerank | Top-K 直接使用向量距离排序 | MRR 仅 0.40-0.50 |
| 无查询改写 | 口语化问题直接检索 | 检索命中率下降 20-30% |
| 高幻觉率 | 检索不相关时 LLM 强行编造 | 幻觉率 15-25% |

**示例**：用户问「去年 Q3 的营收是多少？」，文档中原文是「2025 年第三季度实现营收 1.2 亿元」。Naive RAG 因「去年」与「2025 年」的 Embedding 相似度不足，可能检索到不相关的财务文档，LLM 最终编造了一个数字。

## 三、Advanced RAG 关键技术

### 3.1 技术全景表

| 技术 | 原理简介 | 性能提升数据 | 代表实现 |
|------|---------|------------|---------|
| **混合检索（Dense + Sparse）** | Dense Embedding 捕获语义相似度，Sparse（BM25）捕获精确关键词匹配，两路结果融合 | Recall@5 +15-25% | BGE-M3（同时输出 dense + sparse）、Elasticsearch hybrid |
| **RRF 融合排序** | Reciprocal Rank Fusion：对多路检索结果按排名倒数加权融合，公式 `score = Σ 1/(k + rank)`，k 通常取 60 | MRR +8-12% | LangChain EnsembleRetriever、自定义实现 |
| **Cross-Encoder Reranker** | 将 Query + 每个候选 Chunk 拼接后输入 Cross-Encoder 模型，输出相关性分数（0-1），重排 Top-K 结果 | Precision@5 +20-35% | BGE-Reranker-v2-m3、Cohere Rerank、Cross-Encoder/ms-marco-MiniLM |
| **句子窗口检索** | 检索到某个句子后，返回该句子所在的「窗口」（前后 N 句），而非仅返回该句子本身 | 上下文完整性 +30% | LlamaIndex SentenceWindowNodeParser |
| **元数据过滤** | 在检索前先根据元数据（日期/作者/分类/版本）缩小搜索范围，再执行向量检索 | 检索精度 +15-20%，延迟 -40% | ChromaDB where 过滤、Qdrant payload filter |
| **Query Rewriting** | 用一个小模型（或 LLM）将用户口语化问题改写为更规范的检索 query | Recall@5 +10-15% | HyDE (Hypothetical Document Embeddings)、Rewrite-Retrieve-Read |
| **HyDE** | 先让 LLM 根据用户问题生成一个「假想文档」，再用该文档的 Embedding 去检索真实文档，解决 query-document 语义 gap | 跨域检索 Recall +20% | HyDE 论文（Gao et al., 2022）|

### 3.2 典型管线示例

```
用户输入
    │
    ▼
Query Rewriting（LLM 改写 → 标准检索 Query）
    │
    ├──→ Dense Retrieval（BGE-large-zh-v1.5）──┐
    │                                           │
    ├──→ Sparse Retrieval（BM25 + jieba）──────┼──→ RRF 融合 → Cross-Encoder Reranker → Top-3
    │                                           │
    └──→ 元数据过滤（日期范围 + 分类）──────────┘               │
                                                                ▼
                                                          LLM 生成回答
```

### 3.3 关键技术选型注意事项

- **RRF 的 k 值选择**：k=60 是工业界默认值（来自原始论文），但实际项目中 k 值需要根据数据分布调整。实测发现当检索结果数量 < 10 条时，k 取 30 效果更好。
- **Cross-Encoder 的延迟成本**：对 Top-50 做 Rerank 需要 50 次模型推理，单次约 5-10ms，总计 250-500ms。如果对延迟敏感，需要降低候选集大小或使用更小的模型。
- **HyDE 的适用边界**：HyDE 在跨领域检索（如医疗问题检索法律文档）效果显著，但在同一领域内效果提升有限（< 5%），引入额外的 LLM 调用成本和延迟。

## 四、Agentic RAG 架构详解

### 4.1 核心思想

Agentic RAG 的核心变化是：**检索不再是 RAG 管线的固定步骤，而是 Agent 的一个工具（Tool）**。LLM 自主决策：

1. **什么时候检索**——不是每个问题都需要检索。简单事实性问题、模型已掌握的知识、用户提供的完整上下文，都不需要触发检索。
2. **检索什么**——检索 query 由 Agent 自主生成，不再是用户原始问题。Agent 可以将复杂问题拆解为多个子问题，分步检索。
3. **怎么用检索结果**——检索到的文档可能部分相关、部分矛盾、部分过时。Agent 需要判断哪些可用、哪些需要丢弃、哪些需要进一步检索验证。

### 4.2 关键技术方案

#### Self-RAG

**原理**：在生成每个片段前，LLM 先判断是否需要检索（retrieval token）。如果需要，检索后对每个检索结果生成「相关性」和「有用性」两个反思标记（reflection tokens），根据反思标记决定是否使用该结果。

**性能数据**：
- 在 6 个公开数据集上，Self-RAG 比标准 RAG 的 Factuality 提升 **12-18%**
- 不必要的检索次数减少 **40-60%**
- 额外延迟：每次生成增加 1 次 LLM 调用（反思标记预测）

#### Corrective RAG (CRAG)

**原理**：检索后先用一个轻量级相关性评估器（如 T5-large）对检索结果打分。如果所有结果相关度均低于阈值（如 0.3），触发「修正机制」——使用 Web 搜索替代知识库检索；如果部分相关，丢弃低相关结果；如果全部相关，正常使用。

**性能数据**：
- 在 PopQA 数据集上，准确率从标准 RAG 的 41.2% 提升至 **58.7%**
- 在低质量知识库场景（检索准确率 < 50%）效果提升最显著

#### Adaptive RAG

**原理**：根据用户问题的复杂度动态选择检索策略。简单问题用单步检索直接回答；中等复杂度问题用多步检索（拆解子问题 + 分步检索）；复杂问题先检索获取背景知识再生成。

**性能数据**：
- 简单问题的回答延迟降低 **50%**（跳过不必要的多步检索）
- 复杂问题的回答准确率提升 **15-22%**
- 总 LLM token 消耗降低 **20-35%**

#### FLARE (Forward-Looking Active Retrieval Augmented Generation)

**原理**：LLM 在生成过程中主动预测「下一句需要哪些知识」，提前触发检索，实现检索与生成的交错执行。每次生成遇到不确定的 token 时，触发检索来辅助生成。

**性能数据**：
- 长文本生成（> 2000 token）的 Factuality 提升 **25-30%**
- 每条生成最多触发 3-5 次检索，额外延迟约 2-5s

### 4.3 Agentic RAG 综合架构图

```
用户问题
    │
    ▼
┌─────────────────────────────────────┐
│         LLM Agent（决策中心）          │
│  - 判断是否需要检索                    │
│  - 拆解问题为子问题                    │
│  - 选择检索策略 / 检索源               │
│  - 评估检索结果质量                    │
│  - 决定是否补充检索                    │
└──────┬──────────────────────────────┘
       │
       ├──→ [工具1] 知识库检索（本地 RAG）
       ├──→ [工具2] Web 搜索（Tavily / Bing）
       ├──→ [工具3] 代码执行（Python REPL）
       ├──→ [工具4] API 调用（数据库 / 内部系统）
       └──→ [工具5] 自我验证（LLM 检查生成内容）
```

| **内存占用（10 万条 768 维）** | ~1.2GB | ~0.8GB | ~2.5GB | ~1.5GB | 不适用（托管） |

### 5.2 本项目选择 ChromaDB 的原因

**AI Research Copilot** 当前阶段选择了 ChromaDB，核心理由：

1. **零配置**：`pip install chromadb` 即可使用，无需单独启动服务、无需 Docker、无需配置文件
2. **本地运行**：所有数据存储在本地磁盘，不依赖外部服务，离线可用
3. **Python 原生**：API 设计与 Python 生态无缝集成，支持 `pd.DataFrame` 直接导入，开发者体验优于其他方案
4. **同进程嵌入**：与 FastAPI 应用运行在同一进程，避免网络往返延迟——实测检索延迟比 Qdrant（Docker 部署）低 5-8ms
5. **当前规模适配**：项目当前知识库不超过 10 万条向量，ChromaDB 在此规模下性能完全满足需求

**未来迁移路径**：如果项目知识库超过 100 万向量，迁移计划为 ChromaDB → Qdrant（中等规模）或 Milvus（大规模）。ChromaDB 支持导出为标准格式，迁移成本可控。

### 5.3 选型决策注意事项

- 向量数据库的选择应与项目阶段匹配：**0-1 阶段用 ChromaDB，1-10 阶段用 Qdrant，10-100 阶段用 Milvus**。不要在 0-1 阶段上 Milvus 集群，运维成本会拖慢产品迭代。
- 托管方案（Pinecone）的优势是免运维，但数据主权和隐私合规可能成为问题。对于金融/医疗等合规敏感场景，优先考虑自建方案。

## 六、本项目（AI Research Copilot）的 RAG 方案

### 6.1 三阶段混合检索管线

```
用户查询
    │
    ▼
┌─────────────────────────────────────┐
│  阶段一：Dense + Sparse 并行检索      │
│  ├─ Dense: BGE-large-zh-v1.5 (768维) │
│  └─ Sparse: BM25 + jieba 分词        │
└──────────────┬──────────────────────┘
               │
               ▼
┌─────────────────────────────────────┐

               ▼

└──────────────┬──────────────────────┘
               │
               ▼
          LLM 生成回答
```

### 6.2 实测效果数据

在内部 Golden Dataset（当前 200 条测试用例）上的评估结果：

| 指标 | Naive RAG（基线） | Advanced RAG（当前） | 提升幅度 |
|------|------------------|---------------------|---------|
| Recall@1 | 0.31 | 0.52 | +67.7% |
| Recall@3 | 0.47 | 0.68 | +44.7% |
| **Recall@5** | **0.58** | **0.79** | **+36.2%** |
| **MRR** | **0.45** | **0.67** | **+48.9%** |
| Precision@5 | 0.42 | 0.61 | +45.2% |

## 七、RAG 不应使用的 5 种场景（否定检索测试用例）

RAG 不是万能的。以下场景中，RAG 检索不应被触发，或检索结果不应被使用：

### 场景 1：单篇文档 < 100 字不需要 RAG 检索

文档内容过短（< 100 字）时，Embedding 向量无法捕捉有意义的语义信息，检索结果噪声比远大于信噪比。此类文档应直接拼入 System Prompt 作为上下文，而非经过 RAG 管线检索。

**判断规则**：如果文档字数 < 100 字，跳过 Embedding 和索引步骤，直接作为上下文使用。

### 场景 2：用户问题已经包含完整上下文时不应触发检索

当用户问题本身已经包含了完整的上下文信息（如「根据我们之前讨论的结果，总结一下 Q2 的结论」），RAG 检索的 query 与用户问题相同，检索到的结果反而可能引入不相关的噪声。

**判断规则**：如果用户问题中包含了明确的上下文引用标记（如「如上所述」「根据之前的讨论」「总结一下」「我们」等指代性词），且模型已具备足够的对话历史，应跳过检索。

### 场景 3：检索结果相关度全部 < 0.3 时不应强行注入 Prompt

Cross-Encoder Reranker 输出的相关性分数在 0-1 之间。如果 Top-5 结果中最高分数仍 < 0.3，说明知识库中没有与用户问题相关的信息。此时将不相关信息注入 Prompt，不仅不会帮助 LLM，反而会误导 LLM 编造答案。

**判断规则**：如果 max(relevance_scores) < 0.3，跳过知识库注入，LLM 仅依靠自身知识回答（或告知用户「知识库中未找到相关信息」）。

### 场景 4：实时性要求 < 1 秒的场景不适合三阶段检索（应降级为纯向量检索）

三阶段混合检索（Dense + BM25 + Rerank）的端到端延迟约 300-500ms，无法满足实时性要求 < 1 秒的场景（如客服聊天、实时推荐）。此类场景应降级为纯向量检索（单阶段，仅 Dense Embedding + Top-K），延迟可控制在 20-50ms。

**判断规则**：如果场景的 SLA 要求 P99 响应时间 < 1s，且 RAG 检索的延迟预算 < 100ms，则跳过 BM25 和 Rerank 阶段，仅执行单路 Dense 检索。

### 场景 5：没有标注来源的检索结果禁止直接用于生成医疗/法律/金融建议

医疗、法律、金融等高风险领域要求回答必须有明确的引用来源。如果检索结果没有标注文档来源（原始文件路径、作者、发布日期、版本号），LLM 生成的内容无法追溯，可能导致严重的合规风险。

**判断规则**：对于合规敏感领域的查询，如果检索结果缺少来源元数据（源文档路径、创建日期、作者），系统应拒绝生成并返回提示：「检索结果缺少来源信息，无法提供可信的回答」。
