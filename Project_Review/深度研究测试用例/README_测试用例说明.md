## 如何使用这些测试

### 1. 上传到项目知识库

```bash
# 在 AI Research Copilot 前端上传，或通过 API：
for file in /c/Users/邓凤坤/Desktop/检索知识库/*.md; do
  curl -X POST http://localhost:8000/api/knowledge/upload \
    -F "file=@$file"
done
```

### 2. 按类别逐类测试

建议测试顺序：类别一 → 类别二 → 类别三 → 类别四 → 类别五 → 类别六 → 类别七 → 类别八

**原因**：从简单到复杂，先确认基础检索（精确匹配）OK，再测语义区分，最后测跨文档综合。

### 3. 评估维度

对每个测试用例，记录：

| 维度 | 评分标准 |
|------|---------|
| **召回是否命中** | ✅ 正确答案在 top-3 / ⚠️ 在 top-10 / ❌ 未命中 |
| **排序是否合理** | ✅ 正确答案排第一 / ⚠️ 排 2-3 / ❌ 排在后面 |
| **是否有幻觉** | ✅ 只返回文档中有的信息 / ❌ 编造了文档没有的内容 |
| **干扰排除** | ✅ 没有把无关文档排到前面 / ❌ 无关文档反而排第一 |

### 4. 量化指标

```
整体 Recall@3 = 正确答案在 top-3 的查询数 / 总查询数
整体 Recall@5 = 正确答案在 top-5 的查询数 / 总查询数
整体 MRR = 平均(1 / 正确答案排名)
```

### 5. A/B 对比

改 RAG 参数（embedding 模型、chunk_size、top_k）前后跑同一批测试，对比指标变化，量化优化效果。

## 已知的当前架构弱点（哪些测试可能不通过）

基于对项目代码的分析，以下测试大概率会失败：

| 测试# | 可能失败原因 | 根因 |
|-------|------------|------|
| 3.1 | 精确错误信息匹配不准确 | 纯向量检索，无 BM25 稀疏检索 |
| 4.1 | 跨文档综合不完整 | 检索只在单个 collection 里做，无跨 chunk 合并 |
| 5.1 | 否定逻辑理解偏差 | 向量检索对"不允许"这类否定词不敏感 |
| 6.1 | "赚了多少钱"找不到"营收" | all-MiniLM-L6-v2 对中文同义表达的覆盖差 |
| 8.2 | 极短查询"v3.2"召回质量差 | 短查询语义信息不足 |
| 7.2 | 分不清"规范标准"和"项目实际" | 文档 01 讲的是 bge-m3 标准，但项目实际用 MiniLM |

**这些失败是好事**——它们精确指出了 RAG 优化的方向。修完 embedding 模型 + 加 BM25 + 加 reranker 之后，再跑一遍，观察指标变化。
