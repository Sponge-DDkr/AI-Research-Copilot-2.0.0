### RAG 检索评估维度

对每个测试用例，记录：

| 维度 | 评分标准 |
|------|---------|
| **召回是否命中** | ✅ 正确答案在 top-3 / ⚠️ 在 top-10 / ❌ 未命中 |
| **排序是否合理** | ✅ 正确答案排第一 / ⚠️ 排 2-3 / ❌ 排在后面 |
| **是否有幻觉** | ✅ 只返回文档中有的信息 / ❌ 编造了文档没有的内容 |
| **干扰排除** | ✅ 没有把无关文档排到前面 / ❌ 无关文档反而排第一 |

**量化指标：**
```
Recall@3  = 正确答案在 top-3 的查询数 / 总查询数
Recall@5  = 正确答案在 top-5 的查询数 / 总查询数
MRR       = 平均(1 / 正确答案排名)
```

### 4.2 RAG 检索测试流程

**建议测试顺序：** 类别一 → 类别二 → 类别三 → 类别四 → 类别五 → 类别六 → 类别七 → 类别八

**原因：** 从简单到复杂——先确认基础检索（精确匹配）OK，再测语义区分，最后测跨文档综合和边界。

**测试方式：**
1. 在聊天模式中逐条输入查询
2. 观察 Agent Timeline 中 RAG Trace 卡片——检查检索到的文档、分数、检索管线
3. 记录每条查询的 top-3 结果是否命中、排序是否合理
4. 统计 Recall@3/5 和 MRR

### 4.3 深度研究测试流程

**建议测试顺序：** 快速事实（3.4）→ 技术调研（3.1）→ 市场分析（3.2）→ KB联合检索（3.5）→ 深度综述（3.3）→ 边界压力（3.6）

**测试方式：**
1. 在深度研究模式中输入问题，选择对应的 depth 参数
2. 观察 Agent Loop 执行过程——Plan 是否合理、工具调用顺序、Stop Gate 校验
3. 评估输出报告质量——章节完整性、来源引用、事实准确性
4. 记录每道题的迭代轮数、工具调用次数、总耗时

### 4.4 评估矩阵

| 模式 | 核心指标 | 目标值 | 评估方法 |
|------|---------|:----:|---------|
| RAG 检索 | Recall@5 | ≥ 0.75 | 30 条检索测试的正确答案命中率 |
| RAG 检索 | MRR | ≥ 0.60 | 平均倒数排名 |
| RAG 检索 | 幻觉率 | < 5% | 编造文档不存在内容的占比 |
| 深度研究 | Plan 合理性 | ≥ 85% | 人工评估 Plan 拆分是否合理 |
| 深度研究 | FactCheck 准确率 | ≥ 90% | fact_check 标注的可信度是否与实际一致 |
| 深度研究 | Stop Gate 通过率 | 首轮 ≥ 70% | 首轮即通过 Stop Gate 的任务占比 |
| 深度研究 | 平均迭代轮数 | ≤ 8 轮 | 从任务开始到完成的迭代次数 |

## 五、已知架构弱点

### 5.1 已修复的弱点

| 原弱点 | 状态 | 修复方案 |
|--------|:--:|---------|
| 精确错误信息匹配不准确（测试 3.1） | ✅ 已修复 | 加入 BM25 稀疏检索（jieba 分词），BM25 对精确字符串匹配天然优于向量检索 |
| 中文同义表达覆盖差（测试 6.1） | ✅ 已修复 | Embedding 从 all-MiniLM-L6-v2 升级到 BAAI/bge-large-zh-v1.5（1024维），中文语义理解大幅提升 |
| 分不清"规范标准"和"项目实际"（测试 7.2） | ✅ 已修复 | RAG 检索增加了 reranker 精排阶段，通过 Cross-Encoder 对 query-document 对打分，排序更准确 |

### 5.2 当前仍存在的弱点

| 弱点 | 影响的测试 | 根因 | 优化方向 |
|------|----------|------|---------|
| 跨文档综合不完整（测试 4.1-4.5） | 类别四全部 | 检索在每个 collection 内独立进行，无跨 chunk 合并机制 | 引入 Multi-hop 检索或 Agentic RAG 的迭代检索 |
| 否定逻辑理解偏差（测试 5.1-5.5） | 类别五 | 向量检索对"不允许""禁止""不应"等否定词的语义权重不够 | Embedding 模型对否定词的注意力机制改进，或增加否定关键词的 BM25 权重提升 |
| 极短查询召回质量差（测试 8.2） | 8.2, 8.3, 8.5 | 短查询（1-3个字）语义信息不足，向量检索退化为近似随机 | 对短查询启用 Query Expansion（HyDE 或 LLM 改写） |
| 11 份文档的跨文档排序歧义 | 2.5, 8.4, 8.5 | 文档数量增加后，同名指标（如"毛利率""市场份额"）出现在多份文档中，排序需要上下文 | 引入元数据过滤（领域标签），让用户可以先按领域缩小检索范围 |
| Reranker 延迟 | 所有检索查询 | Cross-Encoder 处理 30 对 query-doc 约 200-500ms，纯 CPU | GPU 加速（CUDA）或模型量化（int8），或对 quick 模式跳过 reranker |

### 5.3 新增文档后的检索挑战

随着文档从 6 份扩展到 11 份，以下新挑战值得关注：

1. **领域混淆**：07(AI编程助手) 和 06(Agent选型) 都涉及 AI 工具对比；08(新能源汽车) 和 03(Q3总结) 都涉及市场和竞品分析——跨领域的检索噪音增加
2. **否定句式激增**：11 份文档共含 60+ 条否定/禁止/条件句式，分布在 8 份文档中，对否定逻辑检索的压力显著增加
3. **数字密度提升**：文档总数字数据点从约 40 个增加到约 150+ 个，精确数字检索的歧义风险上升
